# Database Schema

## Overview
This document defines the Firestore schema for articles uploaded by journalists through the app.
The schema is designed to be consistent with the existing `ArticleEntity` used by the news API, while adding journalist-specific metadata and Firebase Cloud Storage references.

---

## Collection: `articles`

Each document in the `articles` collection represents a journalist-authored article.

### Document ID
Auto-generated by Firestore (`String` — Firestore document ID).

### Fields

| Field            | Type        | Required | Description                                                                 |
|------------------|-------------|----------|-----------------------------------------------------------------------------|
| `id`             | `String`    | Yes      | Firestore document ID (same as the document key)                           |
| `authorId`       | `String`    | Yes      | UID of the authenticated journalist who uploaded the article               |
| `author`         | `String`    | Yes      | Display name of the journalist                                              |
| `title`          | `String`    | Yes      | Title of the article (max 200 characters)                                  |
| `description`    | `String`    | Yes      | Short summary or subtitle of the article (max 500 characters)             |
| `content`        | `String`    | Yes      | Full body text of the article                                              |
| `thumbnailPath`  | `String`    | Yes      | Path reference to the thumbnail image in Firebase Cloud Storage — folder: `media/articles/{articleId}` |
| `thumbnailURL`   | `String`    | Yes      | Public download URL of the thumbnail image stored in `media/articles/`    |
| `category`       | `String`    | No       | Category/tag of the article (e.g., "Technology", "Politics", "Health")    |
| `publishedAt`    | `Timestamp` | Yes      | Timestamp of when the article was published                                |
| `updatedAt`      | `Timestamp` | Yes      | Timestamp of the last update to the article                                |
| `isPublished`    | `Boolean`   | Yes      | Whether the article is visible to all users (`true`) or still a draft      |

### Example Document

```json
{
  "id": "abc123xyz",
  "authorId": "uid_journalist_001",
  "author": "Jane Doe",
  "title": "The Rise of Renewable Energy in 2026",
  "description": "A deep dive into how solar and wind energy are reshaping the global economy.",
  "content": "Renewable energy sources have seen an unprecedented growth over the last decade...",
  "thumbnailPath": "media/articles/abc123xyz/thumbnail.jpg",
  "thumbnailURL": "https://firebasestorage.googleapis.com/v0/b/your-app.appspot.com/o/media%2Farticles%2Fabc123xyz%2Fthumbnail.jpg?alt=media",
  "category": "Technology",
  "publishedAt": "2026-04-09T10:00:00Z",
  "updatedAt": "2026-04-09T10:00:00Z",
  "isPublished": true
}
```

---

## Firebase Cloud Storage Structure

```
media/
  articles/
    {articleId}/
      thumbnail.jpg    ← article thumbnail image
```

The `thumbnailPath` field in Firestore stores the relative path (e.g., `media/articles/abc123xyz/thumbnail.jpg`), while `thumbnailURL` stores the full public download URL used by the Flutter app to display the image.

---

## Design Decisions

- **`authorId` vs `author`**: Both are stored so we can display the author's name without an extra lookup while still being able to query by UID for journalist-specific views.
- **`thumbnailPath` + `thumbnailURL`**: Storing both the storage path (for deletion/update operations) and the download URL (for direct display) avoids redundant Firebase Storage calls on every read.
- **`isPublished`**: Allows journalists to save drafts before making them public, enabling a more complete editorial workflow.
- **`Timestamp` types**: Using Firestore `Timestamp` instead of `String` for dates enables efficient ordering and querying by time.
